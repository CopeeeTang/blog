## 为什么需要这个 skill

做过几个月 Claude Code 之后，我发现一类反复出现的问题：

> 我说「帮我写一个用户注册系统」，claude 做完了，我才发现我其实想要的是「支持邮件验证、手机号可选、不需要密码的无密码登录系统」。

这不是 claude 的错，是我表达的问题——或者更准确说，是我自己都没想清楚。

**Interview Mode** 这个 skill 的核心思路：**在执行前，先做一轮结构化访谈，把需求的模糊地带变成清晰约定**。

## Skill 设计

### 触发方式

```bash
# 在任务描述中加前缀
/interview-mode [你的任务描述]

# 或者在对话中直接说
"先帮我做需求访谈，再开始做：[任务]"
```

### 访谈框架

Skill 内部使用一个固定的问题框架，分三层：

**第一层：What — 目标澄清**
- 这个功能/内容的最终用途是什么？
- 谁是使用者（你自己？还是其他人？）
- 「做完」的标准是什么？怎么判断成功？

**第二层：Constraint — 约束收集**
- 技术栈或格式有什么限制？
- 有哪些「一定不能做」的事？
- 时间/规模上有什么约束？

**第三层：Priority — 优先级排序**
- 如果只能做一件事，最重要的是什么？
- 哪些是 nice-to-have，哪些是 must-have？
- 有什么可以暂时 defer 的？

### 输出结果

访谈结束后，skill 会生成一份简洁的**需求确认书**：

```markdown
## 需求确认：[任务名称]

### 目标
[1-2句话，明确要做什么]

### 约束
- [约束1]
- [约束2]

### 优先级
1. [最重要的事]
2. [次要的事]
3. [可以 defer 的事]

### 不在范围内
- [明确排除的内容]

---
**确认后请回复「开始执行」**
```

## 一个完整的例子

**用户输入：**
```
/interview-mode 帮我建一个笔记系统
```

**Claude 的访谈：**

> 好，在开始前先问几个问题来确保方向正确：
>
> 1. **目标**：这个笔记系统是给你个人用，还是需要支持多人？
> 2. **核心场景**：你最常见的笔记场景是什么——快速捕捉想法、整理资料、写长文，还是都有？
> 3. **技术约束**：有偏好的技术栈吗？还是纯粹看哪个适合？
> 4. **集成需求**：需要和现有工具（比如 Obsidian、Notion）集成吗？
> 5. **「做完」的标准**：最小可用版本需要哪些功能？

**用户回答：**
> 个人用。主要是快速记录灵感和阅读笔记。技术栈无所谓，简单就好。不需要和外部工具集成。最小版本：能写 markdown、能搜索、能打 tag。

**需求确认书：**

```markdown
## 需求确认：个人笔记系统

### 目标
面向个人使用的轻量笔记工具，核心场景是快速记录灵感和阅读笔记。

### 约束
- 技术栈以简单为优先，不引入复杂依赖
- 无需多用户支持
- 无需外部工具集成

### 优先级
1. Markdown 编辑（must-have）
2. 全文搜索（must-have）
3. Tag 系统（must-have）
4. 移动端适配（nice-to-have）
5. 数据同步（defer）

### 不在范围内
- 多用户/协作功能
- 外部 API 集成
- 复杂的权限管理

---
**确认后请回复「开始执行」**
```

## 什么时候用

| 场景 | 建议 |
|------|------|
| 简单明确的任务（"帮我重命名这个变量"） | 不需要 |
| 中等复杂度（"帮我写一个 API"） | 可选，快速访谈 2-3 个问题 |
| 复杂/模糊的任务（"帮我建一个系统"） | 强烈建议 |
| 创意类任务（"帮我写一篇文章"） | 建议，至少确认风格和受众 |

## 一个反直觉的发现

做了这个 skill 之后，我发现**访谈本身有时比执行更有价值**——

很多时候，当我被追问「这个的目标是什么」「做完的标准是什么」，我才意识到我自己都没想清楚。访谈的过程逼我理清了思路，有时候做完访谈我就知道怎么自己做了，甚至不需要让 claude 执行。

这个 skill 与其说是给 claude 用的，不如说是给我自己用的。
