## **[P]** 问题重构：这篇文章到底在解决什么？

如果让我用一句话重新定义这篇文章的核心问题，我会这样说：**怎么让 Claude 在没有人工监督的情况下，长时间地完成一项自主编码任务，并且完成得很好？**

注意三个关键词，每一个都值得单独拆开来看。**无人监督**，不是指模型独自思考几秒钟，而是真正意义上的自我驱动——它需要发现错误、判断方向是否偏离、决定何时该停止迭代、何时该求助。这要求的不只是生成能力，而是一种接近自我纠错的元认知能力，而当前语言模型在这一点上出奇地薄弱。**长时间**，意味着跨越单次上下文窗口的极限。人类工程师可以工作八小时，因为大脑有选择性遗忘和重构的机制；但语言模型的"工作记忆"是有限且线性的，一旦填满，质量会以非线性的方式崩塌。**完成得好**，这三个字看似最简单，实则最复杂——谁来定义"好"？用什么标准？如何在没有人类实时介入的情况下让系统自己知道"还不够好，需要继续迭代"？

原文标题 "Harness Design for Long-Running Application Development" 精准但抽象。我更愿意把它翻译成：当你把一个复杂的全栈开发任务扔给 AI，然后离开去喝咖啡，回来时你期望看到什么？一个能跑的应用，还是一堆半成品代码？

值得注意的是 "Harness" 这个词本身。在英语里它有双重含义：一是框架、脚手架、驾驭工具；二是控制、约束、驯化野性能量的缰绳（马具）。两层含义放在这里都恰如其分——这篇文章设计的既是让 AI 能力得以发挥的框架，也是防止 AI 跑偏的约束机制。Harness 不只是"用好 AI"，更是"驾驭 AI"。

坦白说，这篇文章颠覆了我对"单智能体万能论"的预期。在阅读之前，我隐约期待 Anthropic 会展示某种强大的单模型方案——只要模型足够聪明，给它一个足够长的上下文窗口，它就能独立完成一切。但现实是，即便是最先进的模型，在面对真实世界的开发任务时，仍然需要精心设计的外部结构来弥补其固有的局限性。问题不在于模型不够聪明，而在于"智能"本身需要合适的容器才能稳定释放。

## **[R]** 路径分析：为什么之前的方法行不通？

在进入解决方案之前，有必要理解为什么这个问题这么难。两个核心障碍，表面上相互独立，实则在实践中形成致命的组合。

#### 障碍一：AI 的自我评估不可靠

想象一个场景：你让 Claude 设计一个现代化的仪表盘界面，然后问它做得怎么样。它的回答几乎永远是积极的——即使眼前呈现的是一个典型的 2015 年 Bootstrap 模板，带着那个年代特有的蓝色导航栏和 Arial 字体。模型不是在撒谎，它真的认为这个设计是合格的。问题在于**标准的错位**：模型在评估时使用的罗盘是"是否符合常见的 UI 模式"，而人类期望的是"是否有独特的设计语言和视觉个性"。这两把尺子量出来的结果可以完全不同。

传统的 generate -> self-review -> iterate 循环从根本上有一个逻辑致命伤：评估者和生成者共享同一套价值观和信息来源。当生成者用某种审美标准创作时，它用来自我审查的标准往往是同一套。这不是"没有努力检查"，而是"在错误的罗盘指引下反复修正航向"——迭代越多，离目标越远，但模型却越来越确信自己方向正确。自我评估的可靠性问题，本质上无法通过更多迭代来解决，因为每一次迭代都在同一套偏差系统内部运行。

#### 障碍二：上下文窗口是无声的杀手

上下文窗口的退化是隐蔽的，这正是它危险的地方。它不像程序崩溃那样有明确的报错，而更像温水煮青蛙——性能在不知不觉间滑落，而系统本身察觉不到任何异常。

**上下文丢失**是第一种退化模式。当上下文窗口逐渐填满，模型对早期信息的注意力会悄然衰减。一个完整的全栈项目开发过程中，最初制定的设计原则、技术决策、API 约定，在数万 token 之后已经变得模糊。模型不是故意遗忘，而是注意力机制本身决定了近处的信息权重更高。结果是，任务进行到后半段时，生成的代码开始与前半段的架构决策出现细微的不一致——字段命名风格漂移、组件接口不匹配、错误处理逻辑前后不统一。这些都是上下文丢失在代码层面的具体表现。

**上下文焦虑**是第二种退化模式，更加微妙。当模型感知到上下文窗口即将耗尽，它会出现一种恐慌式收尾的行为倾向——开始写下"TODO: implement later"、跳过原本计划中的模块、用存根函数（stub）替代真实实现，甚至开始生成错误的代码只为了"让任务看起来完成了"。这不是真的做不下去，而是上下文接近极限的统计信号触发了一种类似"该结束了"的行为模式，压过了任务本身的需求。

这两种退化组合在一起尤其致命。上下文丢失让模型失去全局一致性，上下文焦虑让模型在最关键的收尾阶段开始敷衍了事。而这恰恰是一个完整项目交付最需要精确和坚持的时刻。

解法出人意料：**不要压缩上下文，直接清空重来。** 对 Sonnet 4.5，彻底 reset 比精心 compaction 更有效。原因在于 compaction 保留了"我已经工作了很久"这个隐含的统计信号——它压缩了内容，但没有清除上下文耗尽的状态感知。而 reset 通过重新注入任务相关信息，让模型以真正新鲜的状态继续工作。压缩保留了退化的信号分布；清空提供了真正的新起点。

## **[C]** 核心方案：三智能体架构与任务解耦方法论

核心贡献不仅是**架构层**的三智能体，更是**方法论层**的任务解耦思维：面对复杂任务，如何判断哪些职责应该分离成独立角色？

### 架构层 -- 三个角色各司其职

**规划者（Planner）** 是整个系统的想象力来源。它接收用户的简短需求，输出一份详尽的技术规格文档。但它的核心能力不只是展开需求——而是主动将模糊意图转化为精确的工程语言，同时注入 AI 时代特有的功能增益。"做一个任务管理器"在 Planner 手中，会变成"一个带智能优先级分类、自然语言搜索和自动截止日期推断的任务管理器"。Planner 的产出是整个系统的共享语言——Generator 依赖它构建，Evaluator 依赖它判断，Sprint Contract 基于它协商。规格文档的质量直接决定了整个 harness 的上限。

**生成者（Generator）** 是系统的执行者，但执行方式至关重要。关键词是"增量构建"——不是一次性生成完整的应用代码，而是按模块逐步搭建，每完成一个单元就接受评估和验证。技术栈选择 React + Vite + FastAPI + SQLite，不是随意为之，而是在开发效率、部署简便性和 AI 可理解性之间的工程权衡。增量构建的价值在于两点：第一，每一轮的上下文负载都相对较短，避免了上下文耗尽；第二，错误被早期捕获，不会积累到最后才爆发。失败半径被严格限制在单个模块的范围内，这使得整个系统具有了一种工程上的弹性。

**评估者（Evaluator）** 是系统客观性的守护者。它用 Playwright 自动化测试实际运行的应用，而不是对代码进行静态分析。这个区别至关重要：代码可以语法正确但行为错误；Playwright 测试的是行为，而非结构。评估者像一个经过培训的 QA 工程师一样操作应用——点击按钮、填写表单、提交数据、检查结果。它输出的不是"代码逻辑看起来不错"，而是具体可行动的描述："用户登录后跳转到首页的过程中出现了 500 错误"，或者"搜索框在输入中文字符时没有响应"。这种行为级评估，正是弥补自我评估不可靠性的关键机制。

### 方法论层 -- 三个可迁移的解耦原则

三智能体不是针对这个特定问题的临时解法，而是从一个更普遍的方法论中推导出来的。这个方法论可以用三条解耦原则来描述：

**原则一：上下文干扰原则。** 当一个角色需要的上下文与另一个角色产生干扰时，就应该分离。规划者需要全局视野和发散思维；生成者需要专注于当前模块的细节实现；评估者需要对生成过程保持无知，才能做出真正客观的判断。三者的上下文需求不兼容，因此必须分离成独立角色。这个原则可以推广到任何需要创造性与执行性分工的场景。

**原则二：评估独立性原则。** 评估者必须在信息上与生成者隔离，才能保持客观性。这不只是角色分工的问题，而是信息流设计的问题。评估者如果读过生成过程的日志、看过 Generator 的内部推理链，它就会不可避免地受到"启动效应"的影响，倾向于在已有框架内评价，而非以用户视角重新审视。黑盒评估的代价是评估者只能报告症状，无法诊断根因——但这个代价是值得付出的，因为客观性比根因诊断更难获得。

**原则三：失败可控性原则。** 每个角色的失败半径应该被限制在其自身职责范围内。Planner 写出一份糟糕的规格，最坏的结果是浪费一次规划；Generator 一个模块构建失败，只需回退到上一个检查点重试；Evaluator 误判了某个功能，只需重新运行测试。没有任何一个角色的失败会导致整个任务的全局崩溃。这种失败可控性，是让系统可以长时间无人监督运行的底层保障。

**冲刺合约（Sprint Contract）** 在整个架构中扮演着防腐剂的角色。它在每一轮生成开始之前，明确协商当前 Sprint 的范围边界——Generator 应该做什么、不应该做什么，Evaluator 应该检查哪些功能、不应该越界要求哪些额外特性。这不是流程上的形式主义，而是防止两种特有失效模式的关键机制：功能蔓延（Generator 在完成核心任务后不受控地添加新功能）和评估越权（Evaluator 以"这个功能不够好"为由拒绝接受任务，超出了原始规格的约定范围）。AI 系统中，范围蔓延比人类团队中更危险，因为模型没有"这不是我的活"的本能，它会不加判断地在已建立的方向上继续延伸。

> 当一个角色需要的上下文与另一个角色产生干扰时，就应该分离——这是任务解耦的核心判断标准，也是一个可以迁移到任何复杂 AI 任务的普遍原则。

## **[I]** 实践展开：从前端引子到全栈编码

实践路径很聪明——先在相对可控的领域（前端设计）验证方法论，再扩展到更复杂的全栈开发。两个领域的案例揭示了截然不同的失效模式，也因此证明了同一套架构方法论的普遍适用性。

### 前端设计：GAN 启发的循环

前端设计是一个特别有趣的试验场，因为它是纯主观领域。代码的正确性可以用测试来验证，但设计的"好坏"没有客观标准。这正是评估者设计最困难的地方——它需要在主观判断领域建立一套可执行、可重复、可量化的评估协议。

真正的创新在于**评分标准的具体化**——四维评分让模糊的"好不好"变成可量化的反馈，而这四个维度之间存在张力，这种张力本身就是设计的难点：

- **功能性** -- 用户能否完成核心操作。好看但不能用 = 彻底失败。这是门槛标准，必须首先满足，其他维度才有意义。
- **设计质量** -- 整体视觉一致性和独特风格（防止"像每一个网站"的均质化）。高分意味着页面有自己的气质，而非从模板库中拼凑。
- **工艺水准** -- 字体选择与对齐、间距系统的统一性、色彩搭配的协调度。细节即专业度，工艺水准往往是区分学生作品和专业作品的分水岭。
- **原创性** -- 自定义设计决策 vs 套用现成模板。原创性与设计质量存在张力：高原创性但低质量意味着大胆但失控；高质量但低原创意味着精致但平庸。

四个维度之间有优先级层次：功能性是必要条件，设计质量是基础分，工艺水准是加分项，原创性是最终差异化因素。一个功能性满分但原创性为零的设计，在这套框架下不会被判定为成功——因为目标本就不是一个"能用的"界面，而是一个"有个性的"界面。

5 次迭代时，设计从通用 Bootstrap 模板脱离；10 次迭代时，设计开始呈现一致的视觉语言；15 次迭代后，设计演化为具有独特性格的作品。这个渐进式的演化过程，与 GAN 训练的动态惊人相似——生成者不断尝试满足评估者的标准，而评估者的标准本身也在每一轮迭代中被更清晰地定义和传达。

### 全栈开发：代价与收获

全栈开发揭示了单智能体方法最深层的失效原因。单智能体失败不是因为不够努力，而是因为它试图在单次输出中解决所有层次的问题：前端组件树的结构、后端 API 的端点设计、数据库 schema 的字段定义，这三层需要彼此一致，但在单次大输出中，它们往往对不上。前端期望的接口字段和后端实际返回的字段不匹配，数据库 schema 定义的类型和 API 层传递的类型不一致。层与层之间的接口错位，是单智能体全栈开发最常见的失效模式。

三智能体 Harness 之所以解决了这个问题，根本原因在于 Planner 为整个系统建立了共享语言。Generator 在构建前端时，参照的是 Planner 规格中明确定义的 API 合约；在构建后端时，参照的是同一份合约。两个层次的构建使用同一张地图，接口错位的问题在源头被消除，而不是在集成阶段才被发现。

### 成本对比

| | 单智能体 | 三智能体 Harness |
|---|---|---|
| **成本** | $9 | $200 |
| **耗时** | 20 分钟 | 6 小时 |
| **结果** | 核心功能未实现 | 功能完整、可交付 |

**不是贵 22 倍，而是从 0 到 1 的区别。**

#### 注意要点

- 增量构建降低认知负担——每完成一模块就接受审查，而非一次性输出。这使得上下文在每一轮保持在可控范围内，同时让 Evaluator 的测试目标清晰聚焦，而非面对一个庞大的整体应用不知从何下手。
- Playwright 真实 UI 测试——评估者像用户一样操作，而非静态代码分析。这意味着测试结果与真实用户体验强相关，而非与代码结构的抽象质量相关。功能是否真正可用，而非功能是否"看起来"正确实现了。
- 冲刺合约防范 scope creep——在实施前协商范围，防止功能蔓延。这在 AI 系统中比在人类团队中更为关键：模型没有"边界感"，它会在已建立的方向上无限延伸，直到某种外部约束介入。
- 角色切换时自然发生 context reset——当 Generator 完成一个 Sprint 并将结果交给 Evaluator 时，新的 Evaluator 会话以全新的上下文启动，只携带 Planner 规格和当前应用状态。架构设计本身就解决了长上下文退化的问题，而不需要任何额外的工程机制。

> 单智能体花了 $9 什么都没交付；三智能体花了 $200 交付了一个功能完整的全栈应用。这不是"贵 22 倍"，而是"从 0 到 1"。两者之间不是价格关系，是可行性关系。

## **[E]** 未来展望：模型能力与 Harness 的动态平衡

随着 Opus 4.6 等更强模型出现，harness 的复杂度可以降低——但降低的速度和方式，因各组件的功能性质而截然不同。

技术演化规律：**工具架构的复杂度应与模型能力呈反比关系。** 今天需要三个智能体协作 6 小时才能完成的任务，明年可能只需要一个更聪明的模型加上一套明确的评分标准。但这个收缩不是均匀发生的，不同 harness 组件的淘汰速度存在显著差异。

最先变得多余的是**评估者**。随着模型输出质量的整体提升，简单任务的第一次生成往往已经足够好，不需要多轮迭代验证。Evaluator 的存在价值依赖于 Generator 的输出质量有较高的方差——当方差收缩，评估者的边际价值自然下降。

其次是**上下文管理机制**。随着模型对长上下文的利用能力提升，对显式 context reset 和增量构建节奏的依赖会减弱。Opus 4.6 已经在这个方向上展现了明显进步，它对上下文新鲜度的感知更准确，不容易在窗口填满时触发焦虑式收尾行为。

然而，**任务分解本身永远不会过时**。任务分解解决的不是能力问题，而是复杂性问题。无论模型多聪明，一个需要同时考虑 UI 审美、后端逻辑、数据库设计、用户体验、性能边界的任务，分解之后依然比不分解更容易把控。复杂性是客观存在的，不会因为单个节点的智能提升而消失。

未来可能的演化路径：3 个智能体 -> 2 个（Planner 与 Generator 合并）-> 1 个更强的通才——但"分解任务、明确标准、迭代改进"的方法论将以更轻量的形式持续存在。这不是 harness 的消亡，而是 harness 内化进了模型本身的能力之中。

更深远地看，这篇文章描述的三智能体架构，可能是我们设计"AI 同事"的第一份组织架构图。当一个 AI 系统可以自主规划、执行、评估、纠错，它在某种意义上已经不再是工具——它是一个有分工、有职责边界、有协作协议的组织。

> 工具架构的复杂度应该与模型能力呈反比关系——模型越强，脚手架越轻。但任务分解的方法论，将以越来越轻的形式永久存在。

## **[+A]** 深度追问：阅读中未被回答的问题

![五个深度追问的可视化概览](/blog/covers/harness-questions.png)
*图 5 -- 深度追问：从上下文管理到架构复用，五个未被原文回答的关键问题*

### 追问 1：Context Reset vs Compaction -- 为什么"丢弃"比"压缩"有效？

这个问题表面上是关于信息量的，但实际上关于的是**分布**。Compaction 解决的是信息量问题：把 100k token 的历史压缩成 10k，让模型能继续工作。但它没有解决分布问题：一个经过 5 小时高强度工作的模型，其内部状态的统计分布，与一个刚刚开始全新任务的模型是不同的。Compaction 保留了"我已经工作了很久"这个隐含信号，以及随之而来的上下文焦虑模式。Reset 则通过重新注入任务相关信息，使模型的输入分布回归到接近全新对话的状态，从而从根本上切断了焦虑信号的来源。

用一个类比来说明：Compaction 相当于把凌乱的桌面整理了一下——东西少了，但你还是在同一张桌子上，仍然带着过去几小时积累的疲劳感。Reset 相当于搬到一张干净的新桌子，只带上手头这个任务必需的三到五份材料——虽然丢弃了很多信息，但工作状态是真正新鲜的。

Opus 4.6 在模型层面对这个问题做了部分改善。它发展出了更强的"上下文新鲜度感知"能力，即使面对接近满载的上下文窗口，也不容易触发恐慌式收尾行为。这意味着它在一定程度上内化了 harness 本来要显式处理的上下文焦虑问题——这确实是 harness 复杂度随模型进步而降低的一个具体案例。但它解决的是焦虑问题，而不是丢失问题：更长的上下文中，早期信息的注意力权重衰减依然存在，只是模型处理这种衰减的方式更优雅了。

### 追问 2：非二元评估的量化方式 -- 代码有 pass/fail，设计质量呢？

我的推断是：评估者使用的很可能**不是**数值评分（比如 1-10 分），而是结构化的自然语言反馈加上阈值判断。原因在于，数值评分对语言模型来说是不透明的行动指南。"设计质量 7/10"传达不了任何可行动的信息；但"间距系统不统一，页面顶部 padding 和卡片内边距使用了不同的基准单位"是具体的、可立即修复的。评估者的价值不在于打分，而在于产生对 Generator 有行动价值的反馈。

四个维度之间存在优先级层次，而非平等加权。功能性是前置条件，不满足则其他维度无意义——这类似于软件测试中的"阻断级缺陷"。设计质量是基础分，工艺水准是精度指标，原创性是最终差异化因素。一个功能满分、设计质量高但原创性为零的方案，在这套框架下会收到"通过，但建议下一轮提升原创性"的反馈，而非被直接拒绝。

这种设计背后有一个更深的洞见：**自然语言是语言模型最擅长的评估协议**。相比数值分数，结构化的文本反馈既利用了模型的核心能力，又天然与 Generator 的工作语言对齐，无需任何格式转换。评估协议应该让模型做它最擅长的事，而不是强迫它用不擅长的格式来表达判断。

### 追问 3：评估者的 Context 隔离 -- 如果评估者读过生成过程，还能客观吗？

理想的评估者输入应该只有两样东西：Planner 产出的规格文档，以及当前运行中的应用 URL。没有源代码，没有构建日志，没有 Generator 的内部推理链。Playwright 的黑盒测试范式在工程层面天然强制了这种隔离——它只能与运行中的应用交互，无法访问代码仓库，因此不存在"读了代码之后再评价行为"的可能。

这种完全隔离有一个明确的成本：评估者只能报告症状，无法诊断根因。"点击保存按钮后数据丢失"是症状；"后端 API 没有正确持久化到数据库"是根因。但这个分工是合理的：症状由评估者报告，根因由 Generator 诊断——因为 Generator 有访问代码的权限。评估者的职责是如实描述用户视角下发生了什么，而不是帮 Generator 想好怎么修。

平衡点的关键词是**"用户级推理"**。一个好的评估者应该能做的，是从症状推断出用户层面可以理解的假设："保存后刷新页面数据消失，可能是前端本地状态未与后端同步，或者后端写入后未正确返回成功状态"。这种推理不依赖于源代码访问权限，但给了 Generator 足够的方向信息来定位问题。这是经验丰富的 QA 工程师的核心能力——不需要看代码，但能提供有价值的诊断方向。

### 追问 4：架构复用为 Skill -- 可共享的是什么，需要定制的是什么？

这套三智能体架构有三个清晰的层次，复用性从高到低递减。**可完全复用的层**是协议和模板：Planner 的规格文档格式、Sprint Contract 的协商模板、Evaluator 的反馈结构。这些是与具体任务无关的工程基础设施，可以直接用于任何类似的代码生成任务。**需要定制的层**是技术约束和业务规则：技术栈选择、项目特定的编码规范、业务领域的约束条件（比如医疗应用的合规要求、金融应用的审计需求）。这一层无法直接复用，但可以通过参数化配置来降低定制成本。

**人类角色的定位**是架构中最核心的设计问题。在这套系统里，人类扮演的是产品经理的角色：定义"什么算好的"，而不是告诉 AI"怎么做好"。这是一个重要的区分——前者是人类的核心判断力，后者是可以委托给 AI 的执行力。人类最不可替代的贡献，是在系统开始之前定义评估标准，以及在系统产出之后判断是否真正达到了目标。这两个时刻，都是 AI 的自我评估最不可信的时刻。

与当前 Agent Teams 架构相比，这套 harness 的独特之处在于两点：一是**结构化的合约协商机制**（Sprint Contract），确保角色之间的任务边界在执行前明确协商；二是**信息流的主动隔离设计**，评估者被架构性地排除在生成过程之外，而不是依赖于角色提示词中的"请保持客观"这种软性约束。这两点是当前 Agent Teams 框架中尚未系统化的能力。

### 追问 5：Opus 4.6 让迭代变多余的能力 -- 具体是哪些能力？

多轮迭代的必要性来自三类问题：**输出质量方差**（第一次生成结果不稳定，可能很好也可能很差）、**全局不一致性**（长任务中前后风格或逻辑不统一）、**注意力盲区**（在生成长输出时对某些细节部分的处理质量显著下降）。迭代的作用就是通过多次尝试，统计性地消除这三类问题的影响。

Opus 4.6 的进步主要体现在：输出质量的方差显著收缩，第一次生成的结果可靠性大幅提升；全局一致性更强，在长任务中能更好地维护早期建立的风格和规则；对上下文中关键约束的注意力更稳定，不容易在高负载时遗漏早期定义的设计原则。这些进步使得对于难度较低的任务，3-5 轮迭代足以完成原本需要 15 轮的任务，甚至一次性生成已经足够好。

但评估者检测能力极限的机制值得单独讨论：**收敛检测**。如果连续 N 轮迭代之后，同一个问题依然出现在评估报告中，且 Generator 的修复方案每次都尝试了不同策略但均告失败，这是强烈信号表明该问题已超出 Generator 当前能力的天花板。此时正确的响应不是继续迭代（因为再多的迭代也无法突破能力天花板），而是改变策略：降低任务要求、拆解成更小的子任务、或者提示人类介入。能够识别"已经迭代足够多次，但这不是迭代能解决的问题"，是一个成熟评估系统的标志。

## 结语

这篇文章让我看到 AI 工程的趋势：**从单打独斗到分工协作，从模糊指令到量化标准，从一次性生成到迭代优化。**

但更让我兴奋的是那些未被回答的问题——它们指向一个更大的研究空间：我们如何为 AI 系统建立可靠的自我治理机制？如何设计信息流的隔离边界？如何让评估协议既客观又对执行者有行动价值？

当 AI 可以自主规划、执行、评估、纠错，它就不再是"工具"——它是"同事"。这篇文章描述的三智能体 harness，可能就是我们设计 AI 同事的第一份组织架构图。

**这不只是工程优化，这是 AI 从"工具"走向"同事"的关键一步。**
